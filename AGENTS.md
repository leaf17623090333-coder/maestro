# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Project Is

maestro is an MCP plugin and CLI tool that provides agent-optimized development orchestration -- structured memory, workflow guardrails, and task lifecycle management for AI coding agents. It manages a 6-stage pipeline (discovery, research, planning, approval, execution, done) with 26 MCP tools and 70+ CLI commands.

## Tech Stack

- TypeScript (strict mode, ES2022 target)
- Bun (runtime, package manager, test runner, bundler)
- MCP SDK (`@modelcontextprotocol/sdk`) for tool server
- citty for CLI command framework
- simple-git for git operations
- zod for schema validation
- Clean architecture: ports (interfaces) -> adapters (filesystem) -> use-cases -> commands/handlers

## How to Build, Test, and Run

```bash
bun install                    # install dependencies
bun run build                  # build MCP server, CLI, hooks, standalone binary
bun test                       # run all tests (targets src/)
bun test src/__tests__/unit/foo.test.ts  # run single test file
bun test --watch               # watch mode
bun run typecheck              # tsc --noEmit
bun run check                  # typecheck + test (CI gate)
bun run dev                    # run CLI from source (bun src/cli/index.ts)
```

Build outputs: `dist/server.bundle.mjs` (MCP server), `dist/cli.js` (npm bin), `dist/maestro` (standalone binary), `hooks/*.mjs` (5 Claude Code hooks).

## Tooling

- **bun**, not npm/yarn/pnpm -- for install, build, test, and runtime
- **build.ts** is a custom Bun build script, not a config file -- it syncs version, generates registries, bundles server/CLI/hooks, compiles binary
- Two auto-generated registries: `src/cli/registry.generated.ts` (CLI commands), `src/skills/registry.generated.ts` (built-in skills) -- regenerated by build
- All three bundles (server, CLI, hooks) externalize `simple-git`

## Rules

- Never edit `*.generated.ts` files directly -- they are regenerated by `bun run build`
- All state mutations use atomic file I/O (`writeJsonAtomic` in `src/core/fs-io.ts`) -- maintain crash safety
- New CLI commands go in `src/cli/handlers/{domain}/` following existing patterns; new MCP tools go in `src/mcp/handlers/`
- Port interfaces define the domain contract; adapters implement them against the filesystem -- do not bypass ports
- Task state transitions are validated in `src/tasks/transitions.ts` -- all 6 states (pending, claimed, done, blocked, review, revision) have explicit allowed transitions
- Run `bun run check` before committing
