# Team Mode Protocol

## Overview

Team mode uses agent delegation to parallelize task execution. You (the skill runner) become the orchestrator. Workers (kraken/spark) handle implementation. Unlike parallel mode (which uses ephemeral sub-agents per wave), team mode maintains persistent workers that pull tasks from the maestro task queue.

## When to Use

- Feature has 4+ tasks
- Tasks are complex enough to benefit from dedicated workers
- Tasks have moderate dependencies (workers can wait for blockers to clear)
- You want persistent workers that self-assign work rather than orchestrator-dispatched waves

**Do not use** when:
- Feature has fewer than 4 tasks (single-agent is more efficient)
- All tasks are tightly sequential (workers will idle waiting)
- Runtime does not support agent delegation

## Prerequisites

Your runtime must support agent delegation (teams, subagents, or equivalent).
- Claude Code: enable Agent Teams via `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`
- Other runtimes: use your native delegation mechanism

## Setup

### 1. Create Team

Create a worker team named "implement-{feature-name}" (description: "Implementing feature: {feature_description}"). Use whatever team/delegation API your runtime provides.

### 2. Sync Tasks

Ensure tasks have been generated by calling `maestro_tasks_sync` (if not already done). Call `maestro_task_list` to see all tasks and their dependency relationships.

For each task, the compiled spec from `maestro_task_next` provides:
- **Subject**: Task title and slug
- **Description**: Context, spec reference, workflow (TDD or ship-fast), expected files to modify, and acceptance criteria (tests pass, coverage meets threshold, no lint errors)

**Dependency example:**
```
Phase 1:
  01-create-schema: Create database schema      (no deps)
  02-create-model: Create model layer           (blocked by 01)
  03-seed-data: Create seed data script         (blocked by 01, independent of 02)

Phase 2:
  04-api-routes: Create API routes              (blocked by all Phase 1)
  05-add-auth: Add authentication               (blocked by 04)
```

Task 01 starts immediately. Tasks 02 and 03 can run in parallel once 01 completes. Phase 2 waits for all of Phase 1.

**BR mirroring**: If `.beads/` exists and `feature.json` has `beads_epic_id`, the orchestrator also mirrors task state to BR. Workers do NOT interact with `br` directly -- the orchestrator handles BR state changes after verifying worker output.

### 3. Spawn Workers

Spawn workers based on task count (see Worker Sizing below).

**For TDD tasks** (features, new code):

Spawn a TDD worker (kraken) with the following prompt:

```
You are a TDD implementation worker on team 'implement-{feature-name}'.

Your workflow:
1. Call maestro_task_next to find the next runnable task
2. Call maestro_task_claim to claim it (transitions to claimed)
3. Read the compiled task spec
4. Follow TDD: write failing tests, implement to pass, refactor
5. Run tests and verify they pass
6. Call maestro_task_done with a summary of changes
7. Call maestro_task_next to find the next runnable task
8. If no tasks available, notify the orchestrator and wait

Project context:
- Feature plan: .maestro/features/<feature-name>/plan.md
- Memory: .maestro/memory/ and .maestro/features/<feature-name>/memory/

Important constraints:
- Do NOT commit -- the orchestrator commits after verification
- Do NOT modify .maestro/ files directly
- Do NOT run br/bv commands
- Report blockers immediately via maestro_task_block rather than guessing
```

**For quick-fix tasks** (bugs, small changes):

Spawn a quick-fix worker (spark) with the same structure but without strict TDD requirement. Spark workers implement first, then add tests.

### 4. Worker Sizing

| Feature Tasks | Workers | Types | Rationale |
|---------------|---------|-------|-----------|
| 1-3 | 1 kraken | Single worker | Overhead of team setup exceeds benefit |
| 4-8 | 2 (1 kraken + 1 spark) | Mixed team | Kraken handles complex tasks, spark handles simple ones |
| 8+ | 3 (2 kraken + 1 spark) | Full team | Two krakens for parallel complex work |

**Never spawn more than 3 workers.** Beyond 3, orchestration overhead dominates and workers contend for files.

## Orchestrator Responsibilities

The orchestrator (you) does NOT write code. You manage workers, verify output, commit changes, and handle blockers.

### Monitor Progress

After spawning workers, periodically check `maestro_status` or `maestro_task_list` for task progress.

**Monitoring cadence:**
- Check task list after each worker reports completion
- If no worker reports for an extended period, check for stuck tasks (tasks in `claimed` state for too long)
- Watch for workers that claim a task but don't progress

**Stale claim handling**: Claims auto-expire after the configured timeout (default 120 minutes). When `maestro_task_next` is called, expired claims are reset to `pending`, making the task available for other workers.

**BR supplementary monitoring**: If `beads_epic_id` exists, use `br ready --parent {epic_id} --json` alongside `maestro_task_list` to see which BR issues are unblocked. This supplements (does not replace) maestro task monitoring.

### Verify Completed Tasks

When a worker marks a task as `done`:

1. **Read the files they changed** -- review for correctness, style, spec compliance
2. **Run the test suite**:
   ```bash
   CI=true {test_command}
   ```
3. **If verification passes:**
   - Commit the changes:
     ```bash
     git add {changed_files}
     git commit -m "{type}({scope}): {description}"
     ```
   - **BR mirror**: If `beads_epic_id` exists:
     ```bash
     br close {issue_id} --reason "sha:{sha7} | verified" --suggest-next --json
     ```
4. **If verification fails:**
   - Call `maestro_task_block` with the failure details
   - The worker can then address the issue
   - Include the error output in the block reason

### Handle Blockers

If a worker calls `maestro_task_block`:

1. Read the blocker reason from `maestro_status`
2. Classify the blocker:

| Blocker Type | Action |
|-------------|--------|
| Missing dependency from another task | Check if the dependency task is done; if not, tell worker to wait |
| Unclear spec | Ask user for clarification, call `maestro_task_unblock` with the decision |
| Tech decision needed | Make the decision (or ask user), record via `maestro_memory_write`, call `maestro_task_unblock` |
| Environment/tooling issue | Diagnose and fix yourself, then call `maestro_task_unblock` |
| Design flaw in plan | STOP all workers, re-evaluate with user |

3. Resolution: Call `maestro_task_unblock` with the decision to resume the task (transitions from `blocked` back to `pending`).

### Handle Worker Failures

If a worker crashes, times out, or produces consistently broken output:

```
[!] Worker 'kraken-1' failed on task 04-api-routes after 2 attempts.
--> Reassigning to kraken-2.
```

1. **First failure**: Ask the worker to retry with more specific guidance
2. **Second failure on same task**: Reassign to a different worker
3. **Third failure**: Stop and ask user -- likely a spec or design issue
4. **Worker completely unresponsive**: Spawn a replacement worker. The stale claim will auto-expire, making the task available again.

### Task Handoff Between Workers

When a dependency clears and a new task becomes available (via `maestro_task_next`):

1. Check which workers are idle (no active task)
2. If an idle worker exists: the worker calls `maestro_task_next` to self-assign
3. If all workers are busy: the task waits until a worker completes its current task
4. If a blocked task becomes unblocked (via `maestro_task_unblock`): notify the worker that the blocker is resolved

---

## Shutdown

After all tasks are done:

1. Verify all tasks are in `done` state via `maestro_task_list` or `maestro_status`
2. Request shutdown for each worker (e.g., "All tasks complete. You can stop.")
3. Wait for shutdown confirmations
4. Tear down the worker team
5. Proceed to Phase Completion Protocol

## Worked Example: Team Mode Execution

**Feature**: "Add authentication system"
**Tasks** (from `maestro_tasks_sync`): 6 tasks across 2 phases

```
Phase 1:
  01-user-model: Create user model and migration
  02-password-hashing: Create password hashing utility
  03-jwt-service: Create JWT token service (depends on 01)

Phase 2:
  04-login-endpoint: Create login endpoint (depends on 01, 02, 03)
  05-registration-endpoint: Create registration endpoint (depends on 01, 02)
  06-auth-middleware: Add auth middleware (depends on 03)
```

**Setup:**

```
[ok] Created team: implement-auth-system
[ok] maestro_tasks_sync complete: 6 tasks with dependencies
[ok] Spawning workers:
  kraken-1: TDD worker (complex tasks)
  spark-1:  Quick-fix worker (simpler tasks)
```

**Execution flow:**

```
--- Phase 1 ---

kraken-1: maestro_task_next --> 01-user-model
          maestro_task_claim("01-user-model")
spark-1:  maestro_task_next --> 02-password-hashing
          maestro_task_claim("02-password-hashing")
  --> 01 and 02 have no deps, run in parallel

[ok] spark-1: maestro_task_done("02-password-hashing", summary: "4 tests pass")
  Orchestrator verifies: tests pass, code looks good
  git commit -m "feat(auth): add password hashing utility"

spark-1: maestro_task_next --> no runnable tasks (03 blocked by 01). Waiting.

[ok] kraken-1: maestro_task_done("01-user-model", summary: "6 tests pass")
  Orchestrator verifies: tests pass, migration runs cleanly
  git commit -m "feat(auth): add user model and migration"

  --> 03-jwt-service is now unblocked
spark-1: maestro_task_next --> 03-jwt-service
         maestro_task_claim("03-jwt-service")

[ok] spark-1: maestro_task_done("03-jwt-service", summary: "5 tests pass")
  Orchestrator verifies: OK
  git commit -m "feat(auth): add JWT token service"

--- Phase 1 Complete ---
Full test suite: 15 tests pass
Phase completion verification: OK

--- Phase 2 ---

kraken-1: maestro_task_next --> 04-login-endpoint
          maestro_task_claim("04-login-endpoint")
spark-1:  maestro_task_next --> 06-auth-middleware
          maestro_task_claim("06-auth-middleware")
  --> 05 is also available, but spark-1 chose 06 first.

[!] spark-1: maestro_task_block("06-auth-middleware", reason: "JWT_SECRET not defined in test environment")
  Orchestrator: Added JWT_SECRET to .env.test
  maestro_task_unblock("06-auth-middleware", decision: "Added JWT_SECRET to .env.test")
  spark-1 retries 06-auth-middleware

[ok] kraken-1: maestro_task_done("04-login-endpoint", summary: "8 tests pass")
  git commit -m "feat(auth): add login endpoint"

kraken-1: maestro_task_next --> 05-registration-endpoint
          maestro_task_claim("05-registration-endpoint")

[ok] spark-1: maestro_task_done("06-auth-middleware", summary: "4 tests pass, after retry")
  git commit -m "feat(auth): add auth middleware"

spark-1: maestro_task_next --> no more tasks. Waiting.

[ok] kraken-1: maestro_task_done("05-registration-endpoint", summary: "7 tests pass")
  git commit -m "feat(auth): add registration endpoint"

--- Phase 2 Complete ---
Full test suite: 34 tests pass

--- Shutdown ---
[ok] All 6 tasks done. Shutting down workers.
[ok] Team implement-auth-system disbanded.
```

## Comparison: Parallel Mode vs. Team Mode

| Aspect | Parallel Mode | Team Mode |
|--------|--------------|-----------|
| Worker lifecycle | Ephemeral (per wave) | Persistent (full feature) |
| Task assignment | Orchestrator assigns per wave | Workers self-assign via `maestro_task_next` |
| Isolation | Git worktree per sub-agent | Shared worktree |
| Commit flow | Merge worktrees, then commit | Orchestrator commits after each task |
| Best for | Medium features, clear independence | Large features, complex dependencies |
| Failure handling | Retry sequentially in main session | Retry with same or different worker |
| Overhead | Worktree creation + merge per wave | Team setup + monitoring |

## Anti-patterns

| Don't | Do Instead |
|-------|-----------|
| Edit code directly as orchestrator | Delegate to workers -- orchestrator manages, never implements |
| Spawn too many workers (>3) | Match worker count to task count (see sizing table) |
| Skip verification | Always verify (read files + run tests) before committing |
| Let workers commit | Orchestrator commits after verification |
| Ignore worker messages | Respond promptly to unblock workers |
| Let workers interact with BR | Orchestrator handles all BR state changes |
| Force workers onto tasks | Workers self-assign via `maestro_task_next` -- guide with priorities, not force |
| Continue after 3 failures on same task | STOP and escalate to user -- likely a spec/design issue |
| Spawn replacement without diagnosing | Understand why the worker failed before spawning another |
