# Workflow Configuration

> Source of truth for how `maestro` work in this repository should be planned, implemented, and verified.
> Generated by `maestro:setup`. Edit directly if the team workflow changes.

## Methodology
Plan-first delivery with approval gates and TDD by default.

1. Discovery: define or refine the feature, gather only the context needed to act, and persist durable findings.
2. Planning: write the execution plan before implementation and make task boundaries explicit.
3. Approval: review the plan, resolve comments, and do not start execution work until the plan is approved.
4. Execution: sync tasks, implement via direct worker CLI launches, and keep reports current.
5. Completion: merge only after verification passes and the resulting state is documented.

### TDD Lifecycle
1. Red: write or update a failing test that demonstrates the expected behavior
2. Green: implement the minimum change needed to pass
3. Refactor: improve structure while keeping tests green
4. Re-run the affected verification commands before merging

## Quality Targets
- Test coverage: no repo-wide numeric threshold is configured, but every behavior change should have automated test coverage or a documented reason it does not
- CI-aware: GitHub Actions runs `bun install`, `bun run build`, and `bun test` on pushes and pull requests to `main`

## Quality Gates Checklist
- [ ] New or changed behavior has corresponding tests
- [ ] `bun run build` passes
- [ ] `bun test` passes
- [ ] CLI text and JSON output remain intentional and backwards-compatible where required
- [ ] No new silent failures or ambiguous errors were introduced
- [ ] Input validation covers user-controlled boundaries
- [ ] No hardcoded secrets, tokens, or credentials were added
- [ ] Documentation is updated when commands, workflows, or public behavior change
- [ ] No unresolved TODO or FIXME markers remain in changed files

## Testing Requirements

### Unit Tests
- Pure logic and parsing utilities
- Dependency-free use cases
- Error paths and boundary cases

### Integration and E2E Tests
- CLI command behavior in temp repositories
- Git flows and CLI command integration
- Filesystem-backed adapters and persisted state
- Plan, task, and merge flows that cross module boundaries

### Manual Verification
- Run the impacted command in dev mode with `bun src/cli.ts <command>`
- When packaging behavior changes, validate the compiled output with `bun run build`

## Commit Strategy
- Frequency: small, scoped commits per completed vertical slice or task
- Format: `{type}({scope}): {description}` using Conventional Commits
- Summary storage: use the commit body and related task or track report to preserve why the change happened

### Commit Types
| Type | Use When |
|------|----------|
| `feat` | New feature or capability |
| `fix` | Bug fix |
| `docs` | Documentation changes only |
| `style` | Formatting only |
| `refactor` | Structural change without behavior change |
| `test` | Adding or updating tests only |
| `chore` | Build, tooling, or infrastructure work |
| `perf` | Performance improvement |
| `simplify` | Meaningful reduction in complexity |

## Repository Workflow Notes
- Start with `maestro status` when working inside an existing tracked feature
- Keep one active implementation thread at a time unless tasks are explicitly parallelizable
- Use `maestro task-start` for managed task execution with session tracking
- Do not merge on a broken baseline; fix failing build or tests before stacking more work
- Update README, AGENTS, CLAUDE, or skill docs when the change materially affects contributor or agent workflows
