# Product Guidelines

## Voice and Tone
Write for technical users and agents with direct, pragmatic, low-ceremony language. Prefer precise statements over marketing language. Errors and help text should explain what happened, why it matters, and what the operator should do next.

## UX Principles
- Optimize for agent readability first, with human-readable text as a close second
- Keep workflows explicit: plan, approve, sync, execute, merge, complete
- Favor stable command names and predictable file locations
- Expose next actions clearly instead of hiding state transitions
- Support automation with structured output while preserving concise terminal UX
- Treat persistent context as a product feature, not incidental documentation

## Branding
Use the product name `maestro` in lowercase when referring to the CLI. Position it as an agent-optimized development orchestrator and plan-first workflow tool. No additional branding constraints are defined in the repo.

## Writing Style
- Use active voice
- Prefer short sentences
- Lead with the outcome or next action
- Avoid jargon unless the audience is expected to know it
- Keep command examples copy-pastable
- Error messages should state the failure, likely cause, and remediation
